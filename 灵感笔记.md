# Test-Agent 灵感笔记

> **用途**：全局视角看 Test-Agent 项目时发现"本来可以怎么设计更好"的沉淀池。
>
> **机制**：宽进窄出。入口宽——什么"更好的设计"都能记；出口窄——要不要采纳由 [00-协作宪章.md](00-协作宪章.md) §0a 落地准则把关。
>
> **不打断当前 fix**——灵感是沉淀，不是行动。
>
> **跨项目蒸馏**：项目内灵感采纳/不采纳判完后, 通用模式入跨项目灵感库 (`D:\项目文件\灵感库\INDEX.md`)。
>
> 详见通用项目协作宪章 §1.7。

---

## 格式

每条灵感按以下格式：

```
### #N · [一句话标题]
- **日期**：YYYY-MM-DD
- **触发**：在做什么时想到的
- **当前实现**：现在项目是怎么做的
- **更好的方案**：[描述]
- **收益**：[落地友好 / 诚实性 / 可维护 / 跨项目复用 / ...]
- **成本/风险**：[实施成本、引入的复杂度、对落地的影响]
- **采纳判断**（待用户决策）：[落地准则 0a 视角下是否值得采纳]
- **跨项目蒸馏**：[采纳后是否值得蒸馏到通用灵感库, 通用模式描述]
```

### #1 · 协作宪章三层结构

- **日期**：2026-05-13
- **触发**：把 TG 云盘宪章 v4.1 + Test-Agent MEMORY 积淀融合时, 发现两个项目各有独家优点但相互不知道
- **当前实现**：通用宪章 + 实例化模板在 `_精髓库/协作宪章/`, Test-Agent 实例化在 `00-协作宪章.md`
- **更好的方案**：已落地为三层结构 (通用 → 模板 → 项目特化)
- **收益**：
  - 跨项目协作元规则统一, 避免每个新项目重写宪章
  - Test-Agent 独家优点 (一文件一审 / 角色分离 / HANDOFF / 数字铺平) 反哺通用层, TG 下次更新可继承
  - 项目特化层只填差异, 不复述通用机制 — 维护成本低
- **成本/风险**：
  - 通用宪章升 v6+ 时所有实例化文件需 review (校验通用宪章 §7 实例化清单)
  - 版本号管理需严格 (项目锁定 v5 还是跟升 v6)
- **采纳判断**：已采纳 (本次会话落地)
- **跨项目蒸馏**：**强烈推荐**。通用模式 = "元规则 + 实例化模板 + 项目特化" 三层. 适用于任何"多项目共享协作准则"场景, 不限于 Test-Agent / TG. 建议蒸馏到 `D:\项目文件\灵感库\INDEX.md` 作"工程模式 · 协作准则" 条目
