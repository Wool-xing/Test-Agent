# Case Study: 文档诚实化 + 防 mock 闭环 (V1.14.0+1)

> **时间**: 2026-05-13 → 2026-05-15
> **范围**: V1.14.0 → V1.14.0+1
> **PR**: #63 / #64 / #65 / #66 / #67 / #68 (合 6 PR)
> **文件改**: 16 个 .md frontmatter + 4 文档 + 5 runtime + 11 文件 / 339 行加
> **验证**: pytest 23/23 pass · L2 selftest --e2e ✓ PASS 8/9 ok · CI 11 必修 ×3 PR 全绿

## 1. 背景

V1.14.0 起步时, README / ROADMAP 与代码实质之间存在 3 类 gap:

| Gap 类型 | 旧表述 | 实际状态 |
|---|---|---|
| Agent 数字 | "16 expert agents" | 16 个 .md 文件, 实装状态不明 (无 frontmatter 标注) |
| Skill 数字 | "33 business skills + 3 meta-skills" | 实测 32 业务 + 3 meta (33 是误数) |
| Rollout 范围 | "类别 3 垂直领域 2 Agent (V1.x rollout)" | rollout 实跨三类别共 6 个 (env-manager + mobile-tester + visual-tester + system-tester + pentest-tester + automotive-tester) |
| Mock 输出 | ROADMAP "防 mock 承诺" 仅文字声明 | expert 部分硬拒已落, **skill 完全无防 mock** — rollout skill 路由走 fallback no-op stdout `[no-op] skill 'X' has no canonical script; documented step recorded` 返 ok=True 假成功 |

用户反复强调: **修复 ≠ 削话术, 终态 = install + tagent demo + 真 LLM-driven agent 跑通**。

## 2. 阶段分解 (6 PR)

### Phase 1: agent / skill frontmatter labeling (PR #63 / #64 / #65)

为每个 `agents/*.md` (16 个) 加 `EXPERT_IMPL_STATUS:` frontmatter, 为每个 `skills/*.md` (32 个) 加 `SKILL_IMPL_STATUS:` frontmatter。合法值:

```
production  ← 真 LLM-driven runner (orchestrator/agents/*.py) 已实装
script      ← 真 script-backed (utils/*.py) 已实装
rollout     ← V1.x rollout 待实装
vision      ← V2.x 方法论参考 (仅 skill 有此状态)
```

PR #65 自纠错: 初版 4 个 agent 标注与 runtime/orchestrator/agents/ 实代码不一致 (e.g., env-manager 标 active 但无 agent runner), 多源核对 (runtime/ + ROADMAP + test-lead) 后校正。

**教训**: 标注必读 3 源, 不依赖单源 MEMORY 续点档。已沉淀为通用纪律 `test-agent-multi-source-verify` memory。

### Phase 2: README/ROADMAP 数字诚实化 (PR #66)

| 文件 | 旧 | 新 |
|---|---|---|
| `README.md` L44 | "16 expert agents" | "16 expert agents (5 production + 5 script + 6 rollout — see ROADMAP.md)" |

**关键诚实点**: 旧 "33 skill" 是真错 (实 32, 3 meta 单列), PR #66 一并修。

### Phase 3: 防 mock 闭环代码兑现 (PR #67)

ROADMAP 的「防 mock 承诺」从文字声明转为代码实装。**4-step 闭环**:

```
*.md frontmatter (EXPERT_IMPL_STATUS / SKILL_IMPL_STATUS)
  ↓ parse
registry.CatalogEntry.impl_status            ← 单源 (杜绝 hardcoded dict 漂移)
  ↓ 软警告
router._validate_against_catalog             ← issues + confidence -0.3
  ↓ 硬拒
orchestrator.execute_node                    ← returncode=2 + stderr "未实装"
  ↓
用户路由 rollout/vision/unknown → 收到明确说明, 绝不输出 mock
```

#### 5 文件改

1. **`runtime/registry/registry.py`**: `CatalogEntry` 加 `impl_status` 字段; `_entry_from_file` parse 按 kind 选 `EXPERT_IMPL_STATUS` / `SKILL_IMPL_STATUS` key; `_VALID_IMPL_STATUS = {production, script, rollout, vision}` 其他值降级 `unknown` (fail closed)。
2. **`runtime/orchestrator/adapters/experts.py`**: **移除 hardcoded `EXPERT_IMPL_STATUS` dict** (单源迁移, 杜绝 PR #65 双源漂移教训复发); 加 `_get_impl_status(name, kind)` helper 直读 catalog; `execute_node` expert / skill 双分支统一防 mock — rollout / vision / unknown → `returncode=2` + stderr "未实装"。
3. **`runtime/router/router.py`**: `_validate_against_catalog` 改读 `catalog.lookup().impl_status` (不再 import hardcoded dict); expert / skill 双 kind 同检; rollout / vision / unknown 全标 issue + 降 confidence。
4. **`runtime/tests/test_impl_status_filter.py`** (新, 13 cases): registry 无 unknown / counts 校验 / router flag 4 类 + production 不误标 / orchestrator hard rc=2 (5 路径) + **反例 assert "documented step recorded" not in stdout** 验 rollout skill 不再 no-op 假成功。
5. **`ROADMAP.md`**: rollout-6-expert table row 0 状态 `planned → done`; rollout-16-skill 前置改已完成; 防 mock 承诺节展开为双 layer 实装事实; 进度跟踪 +V1.14.0+1 (2026-05-15)。

#### 范围修订过程

初版 X4 task 写 "router 按 EXPERT_IMPL_STATUS 过滤" — 探源后发现:
- expert 部分已在 X1.5 时落地 (hardcoded dict in experts.py L50-67)
- **skill 完全无等价机制** — 16 个 rollout skill (pentest-* 7 / automotive-* 5 / mobile/visual/system/eval-harness 4) + 2 vision skill 走 fallback no-op = **假成功 mock**
- registry 不读 frontmatter → hardcoded vs .md 双源漂移风险

X4 真正核心 = **skill 防 mock + 单源化**, 不是单纯 router 过滤。

### Phase 4: stub 设计不一致根治 (PR #68)

X4 验时发现 pre-existing baseline 坏测试 `test_router_ends_with_report_generator`。根因: V1.14 把 `test-lead` 加到 stub 的 web-system path 末 (按主宪章 §40 "测试主管 — 协调 + 最终上线建议"), 但 mobile-app / desktop-app / ai-model / rest-api 4 path 未同步。

修法: 5 path 末统一 `test-lead` 决策 (与 `agents/README.md` L20-31 流程对齐):

```
bug-manager → report-generator → test-lead 决策
```

测试同步改: `test_router_ends_with_report_generator` → `test_router_ends_with_test_lead_decision`, 末 assert `test-lead`, 末-1 assert `report-generator`。

## 4. 决策记录

### 决策 1: skill 防 mock 是 X4 真核心 (而非 router 过滤)

X4 范围修订: 探源后发现 expert 已部分落地, skill 完全无防 mock。修正 X4 task list, 把 skill execute_node 防 mock 列为优先项。

### 决策 2: 单源迁移 (registry catalog) vs 保留 hardcoded shim

考量 PR #65 自纠错教训 (hardcoded vs frontmatter 漂移)。选 **单源迁移** — registry 直读 frontmatter, 移除 hardcoded `EXPERT_IMPL_STATUS` dict。中间态 (experts.py 改后 router.py 改前) router 短暂失 status check, 但同 PR 内全修, push 前完整。

### 决策 3: X4.5 (production skill no-op) 重审撤回 = NOT-A-BUG

production skill 无 SCRIPT_MAP 仍走 `no-op` 是 **by design**:
- skill = LLM 知识层 (markdown 文档), 非 orchestrator-runnable script
- production skill 路由 = 路由策略告诉 LLM "此步用此 skill", LLM agent 内部消费
- selftest --e2e DAG 9 节点全 expert, 无 skill 节 → skill 顶层 dispatch 仅真 LLM mode + 用户 explicit 路由触发
- ok=True + stdout `[no-op] ... documented step recorded` 是诚实状态描述

f6 假阳性 3 问全否后撤项。

### 决策 4: stub 5 path 统一 test-lead (而非容错测试)

`test_router_ends_with_report_generator` 旧测试错, 不是 stub 错。改修 stub (V1.14 漏改 4 path 同步) + rename 测试, 与 `agents/README.md` 流程对齐。

## 5. 教训 / 复用 pattern

1. **多源核对纪律** — 单源 MEMORY 不足, 必读 3 源 (runtime + ROADMAP + agents 实代码)。沉淀: `test-agent-multi-source-verify`。
2. **单源化优于双源** — hardcoded dict 与文档 frontmatter 双源必漂移, 一律迁单源。registry catalog 单读 frontmatter, downstream 不复制。
3. **一文件一审 + f1-f6** — 协作章程实战: X3 4 文件 + X4 5 文件全停点审, f5 (置信度 H/M/L) + f6 (假阳性 3 问) 自审。f6 在 X4.5 撤回时起决定作用。
4. **范围修订要敢承认错估** — X4 初版 task 写"router 过滤" 不准, 探源后发现 skill 才是真 gap, 直接修订 task list 而非按错 task 走。
5. **诚实化 ≠ mock 修复, 二者递进** — Phase 1 (标注) → Phase 2 (文档) → Phase 3 (代码), 文字诚实是前提, 代码兑现是落地。
