# V2.0.0 技术栈决策

> **日期:** 2026-06-20
> **阶段:** 阶段 1：架构设计
> **状态:** 🔒 已锁定（冻结至发布，除非触发 §零-E 终止评估并选 B）

---

## 决策 1：主语言 → Python 3.10+

| 候选 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **Python** | AI/Agent生态最强（litellm/langchain/pydantic）；现有代码库100% Python；快速原型；丰富测试生态 | 分发复杂（需PyInstaller）；GIL并发受限 | ✅ 选定 |
| TypeScript | 前后端统一；npm生态；Web天然适配 | AI/Agent生态不如Python；现有代码需全重写 | ❌ |
| Go | 性能最优；单二进制分发；并发原生 | AI生态最弱；不适合Agent开发节奏 | ❌ |

**锁定: Python 3.10+**

---

## 决策 2：CLI 框架 → Typer

| 候选 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **Typer** | 现有代码已用；类型提示驱动；Click底层；Rich集成 | 无 | ✅ 选定 |
| Click | Typer基础；成熟稳定 | 需手写装饰器，代码量大 | ❌ |
| argparse | 标准库，零依赖 | 大规模CLI时冗长 | ❌ |

**锁定: Typer (基于现有代码)**

---

## 决策 3：TUI 框架 → 双引擎（prompt_toolkit + Rich + Textual）

| 模式 | 引擎 | 对标 | 用途 |
|------|------|------|------|
| **Chat REPL**（默认） | prompt_toolkit + Rich | Claude Code (Ink/React)、DeepSeek-TUI (Rust) | 自然语言对话、流式Markdown、原生滚动 |
| **Dashboard**（F2切换） | Textual (36K★) | Textualize/textual | 10面板全屏：执行/仪表盘/Skill/配置/日志/调度/状态/帮助/皮肤 |

| 候选 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **prompt_toolkit + Rich** | ~10年成熟（IPython/pgcli使用）；流式Markdown；原生终端滚动+选择复制；Rich 56.5K★ | 不是全屏TUI框架 | ✅ Chat模式 |
| **Textual** | Python原生CSS布局；60fps；35+内置组件；全屏Dashboard | 占全屏丢失原生滚动 | ✅ Dashboard模式 |
| 纯Textual（单引擎） | 统一API | Chat体验差：全屏接管、无原生scrollback、违反Claude Code/DeepSeek体验惯例 | ❌ |
| Ink (Node.js) | Claude Code同款 | 跨语言调用复杂 | ❌ |
| Bubble Tea (Go) | DeepSeek-TUI同类 | 跨语言调用 | ❌ |

**锁定: prompt_toolkit + Rich（Chat模式）+ Textual（Dashboard模式）**

**切换机制:**
- `tagent` → Chat REPL (prompt_toolkit+Rich)
- `tagent tui` → Dashboard (Textual)
- Chat中F2 → 切换到Dashboard
- Dashboard中F1 → 切回Chat

## 决策 4：Web 前端 → React + Vite

| 候选 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **React + Vite** | 现有 apps/desktop-v2/ 路径已用；生态最大；TypeScript友好 | 需单独构建流程 | ✅ 选定 |
| Next.js | SSR/SSG | Test-Agent Web Dashboard不需要SSR；过度复杂 | ❌ |
| Vue + Nuxt | 简单 | 现有代码用React | ❌ |
| 纯HTML | 极简 | 无法支撑Dashboard复杂交互 | ❌ |

**锁定: React 18 + Vite + TypeScript**

---

## 决策 5：数据库 → SQLite (C端) + PostgreSQL (B端可选)

| 候选 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **SQLite** | 零配置；零依赖；嵌入进程；适合单用户 | 不适合高并发写入 | ✅ C端默认 |
| **PostgreSQL** | 生产级；多租户（Schema/DB隔离）；丰富扩展 | 需独立部署 | ✅ B端可选 |
| 纯JSON文件 | 最简 | 无查询能力；不适合大规模数据 | ❌ |

**锁定: SQLite (默认) + PostgreSQL (B端，通过 `mode: enterprise` 切换)**

---

## 决策 6：自测试框架 → pytest

| 候选 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **pytest** | Python标准；现有代码已用；650真实测试已存在 | 无 | ✅ 选定 |
| unittest | 标准库 | 功能弱于pytest | ❌ |
| hammett | 快 | 生态小 | ❌ |

**锁定: pytest + pytest-cov + pytest-xdist (并行)**

---

## 放弃的候选及理由

| 候选 | 放弃理由 |
|------|---------|
| TypeScript为主语言 | 现有31,748行Python代码需全重写；AI/Agent生态不如Python |
| Go为主语言 | AI/Agent库极度匮乏；不适合Agent开发节奏 |
| Next.js | Test-Agent是CLI+TUI为主，Web Dashboard轻量，SSR/SSG不需要 |
| MongoDB | 文档型数据库对测试结果/配置/Skill等结构化数据不友好 |
