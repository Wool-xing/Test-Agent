# 交付物清单

> **本文档回答**：测试计划 / 测试报告 / Bug 列表 / 性能报告 等"对外提交"的产出物**最终落在哪里、由谁负责、什么格式**。
> 全部产出位于 `workspace/` 下，按用途分类如下。

## 2. 三种典型提交场景

### 场景 A：开测前 → 提交"测试计划"评审

**主交付物**：

```text
workspace/测试计划/test_plan_用户登录_20260510.md   ← 主
workspace/需求分析/requirements_analysis_20260510.md  ← 配套（输入分析）
workspace/测试用例/testcases_登录_20260510.xlsx     ← 配套（用例草案）
```

**提交方式**：
- Markdown → 转 PDF/Word 后邮件发送
- 或直接贴 Confluence / Wiki / 飞书云文档
- 或上传 PR 评审

### 场景 B：测试结束 → 提交"测试报告"

**主交付物（1 份 Word + 3 份 HTML 链接）**：

```text
workspace/测试报告/测试报告_20260510.docx       ← 给管理层（Word，含执行摘要+缺陷+性能+风险）
workspace/执行日志/allure-report/index.html         ← 给开发/测试（功能详情）
workspace/执行日志/jmeter-report/index.html         ← 给性能/PM（性能详情）
workspace/执行日志/coverage-report/index.html       ← 给开发（覆盖率详情）
```

**提交方式**：
- Word → 邮件附件
- HTML → CI 部署到 GitHub Pages / 内部静态服务器，发链接（推荐）
- 失败截图 → Word 报告内嵌或 zip 打包附件

### 场景 C：日常 / CI 自动通知

由 `utils/generate_report.send_all_notifications()` 自动推送到企微 / 飞书 / 钉钉群。

通知内容：通过率、Bug 数、性能指标、报告链接。

未配置 webhook 自动跳过，不阻塞流程。

## 3. 报告生成命令

```bash
# Word 报告 + 多端通知（一条命令）
python -m utils.generate_report \
    --data workspace/执行日志/regression_summary.json \
    --notify

# Allure 静态报告（生成）
allure generate workspace/执行日志/allure-results \
    --output workspace/执行日志/allure-report --clean

# Allure 本地服务（临时查看）
allure serve workspace/执行日志/allure-results

# JMeter 报告（执行 JMeter 时已用 -e -o 自动生成）
# 直接打开 workspace/执行日志/jmeter-report/index.html

# 覆盖率报告（pytest 已生成）
# 直接打开 workspace/执行日志/coverage-report/index.html
```

## 4. 完整目录速览（标注交付级别）

```text
workspace/
├── 测试计划/                                  ← 📤 对外（开测前）
│   └── test_plan_{项目}_{YYYYMMDD}.md
├── 需求分析/                                  ← 📤 对外（配套）
│   ├── requirements_analysis_{YYYYMMDD}.md
│   └── requirements_summary_{YYYYMMDD}.json
├── 测试用例/                                  ← 📤 对外（评审 + 配套）
│   └── testcases_{模块}_{YYYYMMDD}.xlsx
├── 测试数据/                                  ← 🔒 内部（含敏感数据，不提交）
│   ├── test_data.json
│   └── jmeter_users.csv
├── 自动化脚本/                                ← 🔒 内部（提交到 Git，但不作交付物）
│   ├── python/
│   └── jmeter/
├── regression_modules.yaml                    ← ⚙️ 配置（可选，提交 Git）
└── 执行日志/
    ├── 报告/                                  ← 📤 对外（最终主报告）
    │   └── 测试报告_{YYYYMMDD}.docx
    ├── allure-report/index.html               ← 📤 对外（功能详情链接）
    ├── jmeter-report/index.html               ← 📤 对外（性能详情链接）
    ├── coverage-report/index.html             ← 📤 对外（覆盖率详情链接）
    ├── allure-results/                        ← 📦 原始（不直接交付）
    ├── jmeter-results/result.jtl              ← 📦 原始
    ├── coverage.xml                           ← 📦 CI 消费（Codecov 上传）
    ├── baselines/perf_baseline.json           ← ⚙️ 内部基线（提交 Git，跨版本对比用）
    ├── history/                               ← 📦 junit-xml 归档（flaky 检测用）
    ├── 截图/                                  ← 📦 调试材料（按需附加到 Bug）
    ├── pytest.log                             ← 📦 调试日志
    ├── 环境检查_{时间戳}.json                  ← 📦 调试材料
    └── junit-results.xml / smoke-results.xml / regression-results.xml  ← 📦 CI 消费
```

**图例**：
- 📤 = 对外提交物
- 🔒 = 内部不提交
- 📦 = 原始 / 调试 / CI 消费
- ⚙️ = 配置（提交 Git）

## 7. CI/CD 自动归档（Artifacts）

GitHub Actions / Jenkins 会自动归档以下产物（无需手工收集）：

### GitHub Actions Artifacts

```text
smoke-test-results        ← workspace/执行日志/（冒烟阶段全部）
regression-test-results   ← workspace/执行日志/（回归阶段全部，含 coverage）
performance-test-results  ← workspace/执行日志/jmeter-results/ + jmeter-report/ + baselines/
```

下载方式：Actions 页面 → 进入对应 run → 底部 Artifacts 区。

### Jenkins Artifacts

archiveArtifacts 已配置：

```text
**/*.xml, **/*.log, **/*.png, **/*.json
+ workspace/执行日志/jmeter-results/**, jmeter-report/**, baselines/**
```

下载方式：构建页面 → Build Artifacts 区。

### Allure 报告（GitHub Pages 自动发布）

main 分支 push 后自动部署到：

```text
https://{用户名}.github.io/{仓库名}/
```

发链接给协作方即可。

## 8. 提交前自检清单

提交测试计划前：
- [ ] `workspace/测试计划/test_plan_*.md` 存在且 12 章节齐全
- [ ] 关联需求分析文档已生成
- [ ] 测试用例 Excel 已生成（4 Sheet）
- [ ] 责任分工与排期与团队对齐

提交测试报告前：
- [ ] `测试报告_*.docx` 已生成（含执行摘要 / 缺陷 / 性能 / 风险）
- [ ] Allure / JMeter / Coverage 三 HTML 报告可正常打开
- [ ] 失败用例已分类（产品Bug / 环境 / 脚本 / Flaky / 数据）
- [ ] P0 Bug 已全部提交 BugTracker（默认禅道,可换）
- [ ] 性能门禁通过（或不通过已说明遗留风险）
- [ ] 覆盖率达标（或不达标已说明）
- [ ] 多端通知已发送（如配置 webhook）

## 9. 历史归档建议

每个迭代版本结束后，建议把以下文件 Git commit 到仓库或归档到内部存储：

```text
workspace/测试计划/test_plan_{项目}_{版本}.md
workspace/测试用例/testcases_{模块}_{版本}.xlsx
workspace/测试报告/测试报告_{版本}.docx
workspace/执行日志/baselines/perf_baseline.json   ← 性能基线必归档（跨版本对比）
```

**不归档**（含敏感或体积大）：
- `workspace/测试数据/test_data.json`（含账号密码）
- `workspace/执行日志/截图/`（体积大）
- `workspace/执行日志/allure-results/` 原始数据（用 allure-report 静态版即可）
- `.env`
