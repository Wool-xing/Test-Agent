# 使用手册

## 🚀 启动指引（首次部署到首次跑通）

> **关键概念**：本仓库是**模板**，需"部署"到你的**真实测试项目目录**才能用。
> 模板目录（`Test-Agent/`）≠ 工作目录（`your-test-project/`）。

### 一图速览

```text
A. 模板仓库（你下载的）          B. 真实测试项目（你工作的）
Test-Agent/    ─部署→  your-test-project/
├── 02-专家定义/                 ├── .claude/agents/
├── 03-技能定义/                 ├── .claude/skills/
├── 04-配置文件/                 ├── conftest.py / pytest.ini / .env / .mcp.json
├── 05-代码示例/                 ├── utils/（78 个 .py）
└── 06-CICD集成/                 ├── .github/workflows/test.yml + Jenkinsfile
                                 ├── src/                ← 你的被测系统源码
                                 └── workspace/          ← 测试产出
```

### 5 步启动

#### Step 1：装外部工具（一次性）

```bash
# 必装
npm install -g @anthropic-ai/claude-code   # Claude Code CLI
# Python 3.10+ / Node.js 18+ / Git（系统包管理器自装）

# 性能测试需要
# Java JRE 17 / JMeter 5.6.3 / Allure CLI 2.27+
```

详见 `部署说明.md` 跨平台安装命令。

#### Step 2：部署模板到测试项目

##### 方式 A：GitHub 一键部署（最快）

仓库根有 `install.sh`，远程一行部署：

```bash
# Mac / Linux（一行远程拉起）
curl -fsSL https://raw.githubusercontent.com/Wool-xing/Test-Agent/main/install.sh | bash -s -- /path/to/your-test-project

# 或先 clone 再本地跑
git clone https://github.com/Wool-xing/Test-Agent.git
bash Test-Agent/install.sh /path/to/your-test-project
```

> 默认仓库为 `Wool-xing/Test-Agent`。fork 后将路径替换为你自己用户名 / 组织名（或用 `TEST_AGENT_REPO_URL` 环境变量覆盖）。
> 私有仓库需先 `gh auth login` 或用 SSH `git@github.com:Wool-xing/Test-Agent.git`。

`install.sh` 自动完成：
1. 检查 git/python3/node/npm
2. clone 模板到临时目录
3. 装 Claude Code（如未装）
4. 创建项目目录结构（含 `测试计划/` 等 13 个 workflow 子目录）
5. 拷贝 16 expert + 32 skill + 78 utils + 配置 + CI/CD 文件
6. 创建 `.venv` + 装 requirements + playwright chromium
7. 复制 `.env.example` → `.env`（已存在则跳过）

##### 方式 B：手动 deploy.sh

详见 `部署说明.md` 完整脚本（含 Windows PowerShell 版）。

#### Step 3：填 `.env`（最少 8 个字段）

```bash
cd your-test-project
notepad .env       # 或 vim / vscode
```

最少必填：

```text
TEST_APP_URL=http://你的测试应用
TEST_API_URL=http://你的测试API
TEST_DB_HOST=...
TEST_DB_PASSWORD=...
ZENTAO_BASE_URL=...
ZENTAO_ACCOUNT=...
ZENTAO_PASSWORD=...
APP_SRC_PATH=./src     ← 指向你的被测系统源码
```

完整字段说明：`配置清单.md`。

#### Step 4：Claude Code 登录 + 启动

```bash
claude /login          # 首次登录（浏览器授权 Anthropic 账户）
cd your-test-project
claude                 # 启动 Claude Code
```

> `ANTHROPIC_API_KEY` **不写入 .env**（避免误提交 Git），由 `claude /login` 管理。

#### Step 5：第一次跑（推荐顺序）

在 Claude Code 提示符内：

```text
> /smoke-test          ← 先冒烟（10 分钟），验证环境与 P0 用例通
```

冒烟通过后再选：

```text
> /test-coordinator    ← 完整流程（含 JMeter 性能）
> /regression-test     ← 回归测试
> /testcase-design     ← 仅生成用例 Excel
```

### 测试工作从哪步开始？

部署 + .env 填好后，按以下推荐顺序：

| 阶段 | 命令 | 目的 |
|------|------|------|
| 0. 自检 | `pytest --collect-only` + `python -c "from utils.api_retry_util import call_with_retry"` | 验证依赖 / utils 可导入 |
| 1. 第一次冒烟 | `> /smoke-test` | 验证环境通、P0 用例通过率 ≥95% |
| 2. 用例设计 | `> /testcase-design 用户登录功能` | 生成 4 Sheet Excel 用例 |
| 3. 脚本生成 | `> /python-script-gen 基于 testcases_login_*.xlsx 生成` | 生成 pytest UI/API 脚本 |
| 4. 完整流程 | `> /test-coordinator 用户登录模块` | 8 专家协同完整流程（含 JMeter） |
| 5. 迭代后回归 | `> /regression-test 对比上版本` | git diff + Flaky + JMeter 基线对比 |
| 6. Bug 提交 | `> /zentao-bug-submission ...` | 规范化提交禅道 |

> 推荐**先 1 → 2 → 3 → 4** 串通整条链路，再 5（回归）和 6（Bug）作为日常补充。

### ⚠️ 常见坑（必看）

| 坑 | 原因 | 解决 |
|----|------|------|
| 直接在模板目录跑 claude 没反应 | `.claude/` 路径不对 | 必须先部署到独立项目目录 |
| 覆盖率永远 0% | `APP_SRC_PATH` 未指向被测系统源码 | `.env` 设 `APP_SRC_PATH=./src`，并把被测代码放进去 |
| `TEST_ENV=prod` 启动 raise | get_current_env 主动拒绝（防误测生产） | 改 `test` / `staging` |
| pytest 报 `Unknown marker: api` | strict-markers + 项目用了未注册 marker | 在 `pytest.ini` `markers` 段补 |
| 性能测试找不到 jmeter 命令 | Java/JMeter 未装或 PATH 没生效 | 见部署说明 JMeter 安装 |
| 通知没发出去 | webhook 未配 | `.env` 填 `WECHAT_WEBHOOK_URL` 等；未配自动跳过不阻塞 |

### 一键自检（部署完跑）

```bash
cd your-test-project
claude --version && python --version && java -version && jmeter --version && allure --version
ls .claude/agents/*.md | wc -l   # 应为 9
```

## 技能（Skill）使用指南（共 32 个：核心 8 + 平台扩展 5）

### `/smoke-test` - 快速冒烟测试

**适用场景：** 代码合并前、每日构建验证、发布前最后确认

```text
> /smoke-test
```

执行内容：
1. 环境健康检查（应用 / DB / Redis / Mock）
2. 仅执行 P0 级用例（不开 reruns）
3. 11 分钟内（含缓冲）给出 通过/阻断 结论
4. **门禁：P0 通过率 ≥95% 且 无新增 P0 Bug**
5. 通过 → 可继续；失败 → 阻断发布

---

### `/test-coordinator` - 完整测试流程

**适用场景：** 新功能上线前、迭代测试、正式测试

```text
> /test-coordinator
> 这次要测试用户模块的登录、注册、密码重置三个功能
```

执行核心 8 位专家 + test-lead（协调者）的完整工作流（按 PRD 自动路由 5 位平台扩展专家）：

```text
需求分析 → 用例设计 → 环境检查 → 数据准备 →
脚本生成（pytest + JMeter JMX） → 冒烟门禁 →
功能回归（P0+P1，含覆盖率） → JMeter 性能测试 →
Bug 管理（BugTracker:默认禅道,可换） → 报告生成（Allure + JMeter HTML + Word + 多端通知:企微/飞书/钉钉/Slack/邮件/Teams）
```

> 注：env-manager 与 data-preparer 严格串行（data-preparer 需要 env 基础 connectivity 通过后才能写 DB）。

### `/regression-test` - 回归测试

**适用场景：** Bug 修复后验证、版本迭代、线上热修复

```text
> /regression-test
> 对比上次版本，本次修复了登录超时 Bug
```

特点：
- 自动分析代码变更影响范围（git diff + `workspace/regression_modules.yaml` 配置化）
- 历史结果对比（从 `workspace/执行日志/history/` 读 junit-xml）
- Flaky 测试检测和隔离（`utils/flaky_detector`）
- JMeter 性能验证 + 基线对比（`workspace/执行日志/baselines/perf_baseline.json`）

## 多格式 PRD 输入（test-lead 自动识别 + 路由）

发 PRD 给 test-lead，**任意格式自动解析 + 平台识别**：

| 格式 | 支持 | 示例 |
|------|-----|------|
| `.md` / `.txt` | ✅ | `> /test-coordinator` + 附 PRD.md |
| `.pdf` | ✅ pdfplumber | `> 这是 PRD: docs/spec_v1.pdf` |
| `.docx` / `.doc` | ✅ python-docx | `> 需求文档：docs/需求.docx` |
| `.xlsx` / `.xls` | ✅ openpyxl 多 Sheet | `> 用例清单 + 需求：docs/spec.xlsx` |
| `.zip`（含多文件） | ✅ 递归解压 + 合并 | `> 完整资料包：docs/spec.zip` |
| `.png` / `.jpg` 截图 | ✅ Claude 视觉 / OCR | `> 这是 UI 设计稿截图：design.png` |
| `.html` | ✅ BeautifulSoup | `> 在线导出页：spec.html` |
| URL（confluence / wiki） | ✅ requests + BS | `> 需求链接：https://...` |

底层：`utils/prd_loader.load_prd(source)` 统一入口。

### 自动平台识别 → 专家路由

PRD 文本中包含的关键词自动识别产品形态，路由到对应专家组合：

```text
PRD 含 "Android" + "iOS" + "REST API" + "1000 QPS"
  ↓
检测到平台：[mobile_android, mobile_ios, api, performance]
  ↓
路由专家：mobile-tester + automation-engineer + automation-engineer(JMeter)
  ↓
test-lead 编排：核心 8 + 上述 3 个分支并行
```

### 直接调用示例

```text
> /test-coordinator
> 需求文档在 docs/PRD_社交APP_v3.0.pdf

[test-lead 自动：]
  1. 调 utils/prd_loader.load_prd('docs/PRD_社交APP_v3.0.pdf')
     → 解析 PDF，提取文本（含表格）
  2. 调 utils.prd_loader.suggest_agents(text)

  4. 输出测试计划（IEEE 829）+ 启动执行链
```

CLI 自检：


## pytest 常用命令

```bash
# P0 冒烟（不开 reruns，配合 flaky_detector）
pytest -m "p0" -n 2 --timeout=60 -v

# P0+P1 并行回归（开 reruns + 覆盖率）
APP_SRC="${APP_SRC_PATH:-./src}"
pytest -m "p0 or p1" -n 4 --reruns=2 --reruns-delay=5 --timeout=120 \
    --cov="${APP_SRC}" --cov-report=html:workspace/执行日志/coverage-report \
    --cov-report=xml:workspace/执行日志/coverage.xml --cov-fail-under=80

# 仅运行登录模块（marker 已在 pytest.ini 注册）
pytest -m "login and (p0 or p1)" -v

# 仅 API
pytest -m "api"

# 查看 Allure 报告
allure serve workspace/执行日志/allure-results

# 调试模式（显示浏览器，跨平台）
# Linux/Mac:
HEADLESS=false pytest -m "p0" -v --no-header
# Windows PowerShell:
$env:HEADLESS="false"; pytest -m "p0" -v --no-header
```

> 覆盖率 `cov` 指向被测系统源码（`$APP_SRC_PATH`，默认 `./src`），不指向测试脚本本身。

```text
> 环境无法访问，如何临时用 Mock 代替？

[env-manager 会提供：]
1. 检查 .env 中的 TEST_APP_URL / TEST_API_URL 是否正确
2. 启动 Mock 服务：MOCK_SERVER_URL（wiremock，docker-compose.test.yml 已含）
3. pytest-mock 创建单元 Mock：fixture(name="mock_user_api", ...)
```

### Q：用例太多，如何快速筛选核心？

```bash
# 仅 P0
pytest -m "p0" -v

# 仅登录模块的 P0+P1
pytest -m "(p0 or p1) and login" -v

# 仅冒烟标记
pytest -m "smoke" -v
```

### Q：如何查看历史测试对比？

```text
> 这次回归测试和上次相比，有哪些新增失败？

[regression-test 自动从 workspace/执行日志/history/ 读取 junit-xml 对比]
```

### Q：Flaky 测试如何处理？

由 `utils/flaky_detector` 自动识别（最近 5 次中既有 pass 又有 fail）：
- 失败率 >30% 标记 `quarantine`（项目应人工加 `@pytest.mark.flaky` 隔离）
- Flaky 用例不计入功能门禁
- 单独 `pytest -m "flaky"` 执行

### Q：如何修改质量门禁？

直接告诉 Claude，它会调整执行策略：

```text
> 这次上线我们可以接受 P1 通过率 90%，请调整质量门禁
```

CI/CD 中：
- 冒烟门禁：`utils/ci_quality_gate.py::GATES['smoke']`（默认 95%）
- 回归门禁：`utils/ci_quality_gate.py::GATES['regression_p0_p1']`（默认 90%）
- 性能门禁：`utils/jmeter_result_parser.py::DEFAULT_GATES_FULL` / `DEFAULT_GATES_CI_QUICK`

直接修改这些常量即可全栈生效。

### Q：性能测试为何 CI 跑不到 100 TPS？

CI 默认 `ci_quick` 模式（5 并发，门禁 TPS≥20）。完整 50 并发压测请：
- 设置 `PERF_MODE=full` 环境变量
- 或在 GitHub Actions workflow_dispatch 手动触发 `release/*` 分支

### Q：基线如何更新？

仅 release 分支 + full 模式 + 全门禁 PASS 时自动更新：

```bash
python -m utils.jmeter_result_parser \
    workspace/执行日志/jmeter-results/result.jtl \
    --mode full \
    --baseline workspace/执行日志/baselines/perf_baseline.json \
    --update-baseline
```

## 最佳实践

### 日常开发流程

```text
每天开始：
  /smoke-test  → 确认环境正常，基础功能可用

新功能开发完成：
  /testcase-design   → 设计用例（4 Sheet Excel）
  /python-script-gen → 生成 pytest 脚本
  pytest -m "p0 or p1"

提测前：
  /test-coordinator  → 完整测试流程

修复 Bug 后：
  /regression-test  → 回归验证（含 JMeter）

发布前：
  /smoke-test  → 最后确认
```

### 团队协作建议

1. `workspace/测试用例/*.xlsx` 和 `workspace/自动化脚本/` 提交到 Git
2. `workspace/regression_modules.yaml` 提交到 Git（回归范围配置）
3. `workspace/执行日志/baselines/perf_baseline.json` 提交到 Git（性能基线）
4. `workspace/测试数据/*.json` **不提交**（含敏感数据 + 频繁变化）
5. `workspace/执行日志/`（除 baselines 外）**不提交**
6. `.env` **不提交**（含密码）
7. Allure 报告通过 CI/CD 自动发布到 GitHub Pages（公网）或内部静态服务器

### 性能优化建议

| 场景 | 建议 |
|------|------|
| 用例执行慢 | 增加并行进程（`-n 8`），不超过 CPU 核数 |
| UI 测试不稳定 | 检查 Playwright 选择器，优先 data-testid > role > label |
| API 测试超时 | 调整 `--timeout=180`，检查环境网络 |
| 数据冲突 | 每个测试用例使用独立数据，通过 `cleanup_tracker` 清理 |
| 性能门禁 CI 不达标 | 切换 `PERF_MODE=ci_quick`（CI 默认）；门禁不严是为快速反馈 |

## V1.1.0 · 运行时 CLI/API 用法

> 运行时层 `runtime/` 是 V1.1.0 新增。已有 16 专家 / 32 Skill / 67 脚本不动,本层仅作可执行调度。

### CLI 命令速查

```bash
python -m runtime.cli.main catalog          # 列 16 专家 + 32 Skill
python -m runtime.cli.main doctor           # 自检 LLM/DB/MinIO/Prefect 可达性
python -m runtime.cli.main plan "<input>"   # 路由,只产 DAG 不执行
python -m runtime.cli.main run "<input>"    # 路由 + 执行
python -m runtime.cli.main run "<input>" --json    # 仅输出执行结果 JSON
```

`<input>` 支持:
- **文件路径**:`./prd.pdf` / `./prd.docx` / `./app.apk` / `./demo.exe`
- **URL**:`https://example.com`
- **自由文本**:`"测试 Web 系统的登录和支付流程"`
- **目录**:`./src/`(源码仓库)

### HTTP API(FastAPI)

```bash
# 起服务
uvicorn runtime.api.main:app --port 8800

# 浏览器开 http://localhost:8800/docs 看交互式 OpenAPI
```

| 端点 | 用途 |
|------|------|
| `GET /health` | 健康检查 |
| `GET /catalog` | 列 16 专家 + 32 Skill |
| `POST /run/text` | 文本输入跑测 |
| `POST /run/file` | 文件上传跑测(multipart) |
| `POST /run/url` | URL 跑测 |
| `GET /status/{run_id}` | 查执行状态 |
| `GET /report/{run_id}` | 查执行结果 JSON |

### 多格式输入示例

```bash
# PDF 需求文档 → 完整测试流程
python -m runtime.cli.main run ./requirements/v2.pdf

# 移动 APK → 自动路由 mobile-tester
python -m runtime.cli.main run ./builds/app-release.apk

# Docker 镜像 tar(docker save 后)
python -m runtime.cli.main run ./images/myapp.tar

# 口头指令
python -m runtime.cli.main run "REST API gRPC 性能测试,QPS 1000"
```
